# T5 — Text-to-Text Transfer Transformer (Google Research, 2019)

## What is T5?
T5 is a transformer model that reframes every NLP task as a text-to-text problem — text goes in, text comes out, for every task without exception. It is an encoder-decoder model, meaning it uses the full vanilla transformer architecture unlike BERT (encoder-only) or GPT (decoder-only). T5's contribution is not a new architecture — it's a unifying framework that changed how the entire field thought about NLP tasks.

---

## The Core Problem T5 Solved
Before T5, every NLP task required a different output format and a different model head bolted onto a pretrained backbone. Classification needed a softmax head. Translation needed a seq2seq setup. Summarization needed its own decoder configuration. QA had yet another format. You couldn't use one model for everything — you had to design, train, and fine-tune task-specific architectures for every problem.

This was inefficient, inconsistent, and didn't scale.

## The Text-to-Text Framing — The Core Insight
T5 says: **every NLP task is just a text-to-text mapping.** Feed text in, get text out. The same model, the same loss function, the same training procedure handles everything.

How it works in practice:

| Task                | Input                                          | Output          |
|---------------------|------------------------------------------------|-----------------|
| Translation         | "translate English to French: Hello"           | "Bonjour"       |
| Summarization       | "summarize: long article text..."              | short summary   |
| Classification      | "cola sentence: The dog ran fast"              | "acceptable"    |
| Question Answering  | "question: What is the capital? context: ..."  | "Paris"         |
| Sentiment           | "sst2 sentence: I loved this movie"            | "positive"      |

The prefix at the start of every input — "summarize:", "translate:", "cola sentence:" — is how T5 knows which task to perform. This is called **task conditioning via prefixes** and it is the direct ancestor of modern prompt engineering. Every time you write an instruction to a language model today, you are using what T5 formalized.

---

## Architecture — Full Encoder-Decoder
T5 uses the complete vanilla transformer: encoder + decoder. This is fundamentally different from BERT (encoder-only, bidirectional) and GPT (decoder-only, causal). The encoder reads and understands the input, the decoder generates the output text autoregressively.

---

## Implementation Changes from Vanilla Transformer

**1. Pre-Norm instead of Post-Norm**
In the original transformer, Layer Normalization is applied *after* the residual connection (Post-Norm). T5 applies it *before* (Pre-Norm). This stabilizes training, especially at scale, by preventing gradient explosion in deep networks. Pre-Norm has since become the standard in almost every modern large language model.

**2. Relative Positional Embeddings instead of Sinusoidal**
The original transformer used fixed sinusoidal positional embeddings that encode absolute position. T5 uses relative positional embeddings — instead of encoding "this token is at position 5," it encodes "this token is 3 positions away from that token." This gives T5 better generalization to sequence lengths it hasn't seen during training and better modeling of relationships between tokens regardless of where they appear in the sequence.

**3. No bias terms in most layers**
A minor but deliberate efficiency choice — T5 removes bias parameters from feed-forward and attention layers, slightly reducing parameter count with negligible performance impact.

---

## Pretraining Objective — Span Corruption
T5 does not use standard MLM like BERT. Instead it uses **span corruption**: random contiguous spans of tokens are masked and replaced with a single sentinel token (like `<extra_id_0>`). The decoder must then predict the original masked spans.

This is more efficient than MLM because the decoder only generates the masked spans, not the entire sequence, making training faster and more focused.

Example:
- Input: `"The <extra_id_0> sat on the <extra_id_1> mat"`
- Output: `"<extra_id_0> cat <extra_id_1> fluffy"`

---

## Scale — The C4 Dataset
T5 was pretrained on **C4 — Colossal Clean Crawled Corpus**, a 750GB cleaned version of Common Crawl. Google spent significant effort cleaning the data — removing duplicate text, filtering non-English content, removing offensive content, and stripping boilerplate. The scale and quality of C4 was a deliberate statement: data quality and quantity matter as much as architecture.

---

## Model Sizes
T5 was released in multiple sizes to show scaling behavior:

| Model    | Parameters |
|----------|------------|
| T5-Small | 60M        |
| T5-Base  | 220M       |
| T5-Large | 770M       |
| T5-XL    | 3B         |
| T5-XXL   | 11B        |

This range allowed Google to systematically study how scale affects performance across tasks — one of the first papers to do this rigorously.

---

## What T5 Did NOT Change
- Core transformer architecture — encoder-decoder, same as the original
- Attention mechanism — still scaled dot-product attention
- The fundamental pretraining then fine-tuning paradigm

---



## The Core Insight
T5's real contribution is philosophical as much as technical. **If you frame every problem the same way, you can solve every problem with the same model.** This unified framing eliminated task-specific architecture design, made transfer learning cleaner, and directly inspired instruction tuning, prompt engineering, and the way every modern LLM is used today. T5 is the conceptual bridge between classical fine-tuning and the prompt-based paradigm.

---

## When to Use T5
Use T5 when your task involves generation — summarization, translation, question answering, or any task where the output is a sequence of text rather than a single label. For pure classification or feature extraction where you need embeddings, BERT or RoBERTa are more appropriate since their encoder-only design is better suited for that.
